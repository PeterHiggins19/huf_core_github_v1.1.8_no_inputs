PR title
feat: HUF coherence adapter ‚Äî retrieval audit for Weaviate exports (namespace dominance + long-tail concentration)
What this adds
A minimal integration example showing how to run a HUF (Higgins Unity Framework) coherence audit on Weaviate retrieval exports. The adapter answers three questions retrieval metrics don't: 1. Which collections / namespaces dominate the result set? 2. How concentrated is the mass? (items_to_cover_90pct) 3. What was discarded ‚Äî and is that acceptable? (error budget) No live Weaviate connection required. Input is a JSONL export.
Demo ‚Äî two commands
# Windows PowerShell
& .\.venv\Scripts\python.exe examples/run_vector_db_demo.py `
  --in cases/vector_db/inputs/retrieval.jsonl `
  --out out/vector_db_demo `
  --tau-global 0.02 `
  --regime-field namespace

& .\.venv\Scripts\python.exe scripts/inspect_huf_artifacts.py --out out/vector_db_demo

# Expected output:
[OK] Wrote artifacts to: out\vector_db_demo
[tail] items_to_cover_90pct=3

Top regimes by rho_global_post:
  1. kb       rho_post=0.619658
  2. tickets  rho_post=0.380342
Why this matters for Weaviate users
Weaviate's namespace and collection model maps directly to HUF's --regime-field. In multi-collection or multi-tenant deployments, one collection can silently dominate retrieval results even when per-collection metrics look healthy. HUF makes that visible: regime ranking, retained-item set, declared discards, and a provenance trace ‚Äî all from a JSONL export. "If the top 3 chunks are stale, 90% of retrieval mass is compromised." That's what items_to_cover_90pct surfaces.
